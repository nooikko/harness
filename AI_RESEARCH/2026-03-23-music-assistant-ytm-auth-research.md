# Research: Music Assistant YouTube Music Authentication

Date: 2026-03-23

## Summary

Music Assistant (the largest open-source home server for streaming music to Cast/Sonos/AirPlay devices) uses **cookie-based authentication exclusively** for YouTube Music, combined with a separate **PO Token Generator** sidecar service. There is **no automated cookie refresh mechanism** — cookies must be manually re-extracted when they expire. The auth architecture has been in near-constant flux since November 2024 when Google removed OAuth, and was significantly disrupted again in March 2025 (PO Token requirement) and October–November 2025 (cookie format changes, yt-dlp BotGuard breakage). These are the exact failure modes relevant to the Harness music plugin.

### 1. Auth Method: Cookie-Only (No OAuth)

**Confidence: HIGH** — from official Music Assistant documentation and source code.

As of **November 2024**, Google removed OAuth from the YouTube Music InnerTube API. Music Assistant's documentation states explicitly:

> "Google has removed OAuth authentication from YT Music — cookie-based authentication is the only way to get YT Music working."

Source: [music-assistant.io/music-providers/youtube-music/](https://www.music-assistant.io/music-providers/youtube-music/)

The three configuration fields required are:
- `Username` — Gmail address or brand account 21-digit ID
- `Login Cookie` — raw browser cookie header string (stored as `SECURE_STRING` in the DB)
- `PO Token Server URL` — URL to the sidecar token generator (default: `http://127.0.0.1:4416`)

Source: `music_assistant/providers/ytmusic/__init__.py`, `get_config_entries()`, confirmed via:
```
gh api repos/music-assistant/server/contents/music_assistant/providers/ytmusic/__init__.py
```

### 2. How Cookies Are Used Internally

**Confidence: HIGH** — directly from source code.

The provider uses **two parallel auth paths** simultaneously:

**Path A — ytmusicapi (for library metadata requests):**
The `_initialize_headers()` method builds request headers with the raw cookie string and derives the `Authorization: SAPISIDHASH ...` value using `sapisid_from_cookie()` and `get_authorization()` from the `ytmusicapi` library. It validates that `__Secure-3PAPISID` is present in the cookie — if missing, raises `LoginFailed("Invalid Cookie detected")`.

```python
sapisid = sapisid_from_cookie(self._cookie)
headers["Authorization"] = get_authorization(sapisid + " " + YTM_DOMAIN)
```

**Path B — yt-dlp (for stream URL extraction):**
The `_get_stream_format()` method converts the cookie to Netscape format via `convert_to_netscape()` and passes it to `yt-dlp` as a `cookiefile`. It forces the `web_music` player client:

```python
ydl_opts = {
    "cookiefile": StringIO(self._netscape_cookie),
    "extractor_args": {
        "youtubepot-bgutilhttp": {
            "base_url": [self._po_token_server_url],
        },
        "youtube": {
            "skip": ["translated_subs", "dash"],
            "player_client": ["web_music"],
        },
    },
}
```

Stream URLs are set to expire after `DEFAULT_STREAM_URL_EXPIRATION = 3600` seconds (1 hour). The URL is re-fetched on each play request, not cached.

The `__Secure-3PAPISID` cookie field is the critical auth token. All library metadata API calls derive their `Authorization` header from it via SAPISIDHASH computation.

### 3. PO Token: Required Since March 2025

**Confidence: HIGH** — from official documentation and multiple GitHub issues.

As of **March 2025**, Google requires a "Proof of Origin" (PO) token for `web_music` client stream access. Without it:

> "Music Assistant will not be able to resolve stream urls for your music without a valid PO Token."

Source: [music-assistant.io/music-providers/youtube-music/](https://www.music-assistant.io/music-providers/youtube-music/)

The PO token is **not generated by Music Assistant itself**. It requires a separate sidecar:

**For Home Assistant addon users:**
Install "YT Music PO Token Generator" from the MA add-on store. It must be running before configuring the YT Music provider.

**For self-hosted/Docker users:**
Run `brainicism/bgutil-ytdlp-pot-provider` Docker image. Music Assistant currently requires **version 1.2.1** (as of docs writing; 1.2.2 was mentioned in issue #4803).

```bash
docker run --name bgutil-provider -d --init brainicism/bgutil-ytdlp-pot-provider
```

Source: [github.com/Brainicism/bgutil-ytdlp-pot-provider](https://github.com/Brainicism/bgutil-ytdlp-pot-provider)

The PO token server runs at `http://127.0.0.1:4416` by default. It uses LuanRT's Botguard interfacing library to generate tokens. Tokens are cached with a configurable TTL (default: 6 hours via `TOKEN_TTL` env var). The token server **automatically refreshes tokens** when the cache expires — this is the only automated refresh in the entire auth stack. Cookies are NOT auto-refreshed.

Source: [github.com/Brainicism/bgutil-ytdlp-pot-provider README](https://github.com/Brainicism/bgutil-ytdlp-pot-provider)

At startup, MA verifies the PO token server is reachable via `GET /ping`. If unreachable, setup fails immediately.

### 4. Cookie Extraction: The Manual Setup Flow

**Confidence: HIGH** — from official documentation.

The documented setup flow for extracting a fresh cookie:

1. Open YouTube Music in an **incognito/private browsing window** (critical — prevents the browser session from being tied to an active logged-in session that could be rotated)
2. Open Developer Tools → Network tab → filter by `/browse`
3. Navigate to an **authenticated library page** (e.g., a private playlist) — this ensures the request is authenticated
4. Find the request in the table; in Request Headers, locate the `Cookie` field
5. Copy the **exact value** — verify it contains `__Secure-3PAPISID`
6. Paste into the Music Assistant provider settings

Source: [beta.music-assistant.io/music-providers/youtube-music/](https://beta.music-assistant.io/music-providers/youtube-music/)

The `ytmusicapi` library documentation (which MA uses for metadata calls) states cookie sessions "remain valid as long as your YTMusic browser session is valid (about 2 years unless you log out)."

Source: [ytmusicapi.readthedocs.io/en/stable/setup/browser.html](https://ytmusicapi.readthedocs.io/en/stable/setup/browser.html)

However, **in practice cookies expire much faster** due to:
- Google rotating sessions on Family/Premium accounts
- Playing music via the web browser on the same account (session takeover)
- VM shutdowns (complete power-off invalidates the session)
- Google's proactive session invalidation for suspected bot activity

Community reports in GitHub Discussion #1551 include users experiencing cookie expiry within **hours to days**, not 2 years. One user: *"I'm so tired of having to replace my Youtube token every day."*

Source: [github.com/orgs/music-assistant/discussions/1551](https://github.com/orgs/music-assistant/discussions/1551)

### 5. Cookie Expiration Handling: Manual Only

**Confidence: HIGH** — confirmed by source code inspection and multiple issue threads.

**There is no automated cookie refresh mechanism anywhere in Music Assistant.**

When the cookie expires, the failure mode is:
- `401: Unauthorized` in logs
- `Unable to fetch PO Token` errors
- `LoginFailed` exception on the next `handle_async_init()` call

The only recovery path is: re-extract the cookie manually from a browser, paste it into Settings → Music Sources → YouTube Music.

There is no:
- Background refresh task
- Session monitoring / health check on a timer
- Cookie validity probe between plays
- Alert/notification when cookie becomes invalid
- Automated re-auth flow

One user in Discussion #1551 explicitly requested: *"Is there any way to detect when the cookie is invalid?"* — suggesting alerts for proactive renewal. The maintainers acknowledged the gap but have not implemented it.

Source: [github.com/orgs/music-assistant/discussions/1551](https://github.com/orgs/music-assistant/discussions/1551)

The MA documentation's guidance for extending cookie lifetime:
- Use a dedicated Google account just for Music Assistant (not shared with any family plan)
- Use incognito/private browsing for extraction (avoids session linking)
- Do not play YTM in the browser using the same account

### 6. Key Auth Reliability Events (2024–2026 Timeline)

**Confidence: HIGH** — from GitHub issue tracker and community forums.

| Date | Event | Source |
|------|-------|--------|
| Nov 2024 | Google removes OAuth from YTM InnerTube API; MA moves to cookie-only | [music-assistant.io docs](https://www.music-assistant.io/music-providers/youtube-music/) |
| Mar 2025 | Google introduces PO Token requirement for `web_music` client streams | [music-assistant.io docs](https://www.music-assistant.io/music-providers/youtube-music/) |
| Oct 2025 | YouTube changes cookie format/exposure order (SID first, not LOGIN_INFO) | [Discussion #4282](https://github.com/orgs/music-assistant/discussions/4282) |
| Oct–Nov 2025 | yt-dlp BotGuard/GVS PO Token breakage; web_music client requires GVS token not provided | [Issue #4339](https://github.com/music-assistant/support/issues/4339), [Issue #5056](https://github.com/music-assistant/support/issues/5056) |
| Nov 2025 | Fix requires Deno runtime in Docker image; MA PR #2636 adds Deno to Dockerfile.base | [Issue #4339](https://github.com/music-assistant/support/issues/4339) |
| Dec 2024 | yt-dlp 2024.12.13 fixes format extraction; MA updates dependency | [Issue #3315](https://github.com/music-assistant/support/issues/3315) |
| Mar 2026 | Fix for yt-dlp stream URL resolution merged (PR #3342 in server repo), targeted for 2.7.11 | [Issue #5056](https://github.com/music-assistant/support/issues/5056) |

MA handles yt-dlp brittleness with a notable architectural decision — **dynamic package installation at startup**:

```python
PACKAGES_TO_INSTALL = ("yt-dlp[default]", "bgutil-ytdlp-pot-provider")

async def _install_packages(self) -> None:
    """Install frequently changing packages dynamically."""
    # NOTE: Google breaks things quite often which requires us to update
    # some packages very frequently. Installing them dynamically prevents
    # us from having to update MA to ensure this provider works.
    for package_name in PACKAGES_TO_INSTALL:
        await install_package(package_name)
```

This means MA always installs the **latest available `yt-dlp`** on startup, not a pinned version. This is an intentional trade-off: accepting yt-dlp API instability in exchange for faster compatibility fixes without requiring a full MA release.

Source: `music_assistant/providers/ytmusic/__init__.py`, `_install_packages()` method

Similarly, v2.7.0 introduced `"YTMusic: Auto update dependencies for frequently breaking packages"` (PR #2311).

Source: [github.com/music-assistant/server/releases/tag/2.7.0](https://github.com/music-assistant/server/releases/tag/2.7.0)

### 7. Premium Account Validation

**Confidence: HIGH** — from source code.

MA validates YouTube Music Premium on every provider initialization using a fixed test video (Rick Astley "Never Gonna Give You Up", `dQw4w9WgXcQ`). It calls `_get_stream_format()` and checks if `format_id == "141"` (256kbps AAC — Premium-only format):

```python
YTM_PREMIUM_CHECK_TRACK_ID = "dQw4w9WgXcQ"

async def _user_has_ytm_premium(self) -> bool:
    stream_format = await self._get_stream_format(YTM_PREMIUM_CHECK_TRACK_ID)
    return stream_format["format_id"] == "141"
```

This check itself is fragile — multiple issues report false negatives (Issue #3408, Discussion #4803) where premium users are rejected because format `141` is unavailable, often due to regional restrictions or yt-dlp version mismatches rather than actual lack of premium.

Source: [github.com/orgs/music-assistant/discussions/4803](https://github.com/orgs/music-assistant/discussions/4803), [github.com/music-assistant/support/issues/3408](https://github.com/music-assistant/support/issues/3408)

### 8. Brand Account Support

**Confidence: HIGH** — from official documentation.

Brand accounts require a **21-digit numeric ID** as the username (found from `myaccount.google.com` top-right account URL), not the email address. The `is_brand_account()` helper detects this and sets `self._yt_user` accordingly.

## Key Takeaways for Harness Music Plugin

1. **Cookie + PO Token sidecar is the definitive architecture.** There is no alternative. OAuth is dead for YTM. Any solution must store a raw cookie string and point at a running bgutil-ytdlp-pot-provider instance.

2. **The PO token sidecar auto-refreshes (6h TTL) — the cookie does not.** The bgutil server handles PO token expiry automatically. Cookie expiry requires human intervention with no current automated path.

3. **Cookie lifetime is unpredictable.** The ytmusicapi docs say "~2 years" but practice shows hours-to-days for accounts shared with browser usage. A **dedicated Google account** for the music plugin is the only reliable mitigation.

4. **yt-dlp is the weakest link.** Google breaks stream extraction multiple times per year. Music Assistant's mitigation is dynamic latest-version installation. Any Harness implementation should similarly avoid pinning yt-dlp and should handle `UnplayableMediaError` gracefully with retry logic.

5. **The `__Secure-3PAPISID` field is the sentinel.** If this is absent from the cookie string, auth fails immediately. This should be validated at configuration time, not at first play.

6. **`web_music` player client is required.** Forcing `player_client: ["web_music"]` in yt-dlp is what enables the music-specific HQ stream formats. Other player clients will either lack Premium formats or require different token flows.

7. **Stream URLs expire in 1 hour.** `DEFAULT_STREAM_URL_EXPIRATION = 3600` — stream details must be fetched fresh per play, not cached long-term.

8. **No multi-browser-tab auth race.** The incognito extraction requirement is specifically to prevent the browser from sending cookie refresh requests in parallel, which can invalidate the extracted session.

9. **Family/Premium shared accounts are unreliable.** Google silently rotates sessions on family plan accounts. The recommended workaround is a standalone paid YTM account.

10. **The premium validation check on `dQw4w9WgXcQ` is itself flaky.** Harness may want to implement a more lenient check (or skip the format ID check and just test for any valid stream) to avoid false-negative premium rejections.
